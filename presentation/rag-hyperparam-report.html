OpsWarden · RAG 超参数调优实验

目标：在误命中率 FPR ≤ 5% 约束下，最大化知识库召回 Hit@1　·　评测集：exact + paraphrase + hard + negative　·　向量库：PostgreSQL / pgvector

模型 BGE-small-zh-v1.5
维度 512
调参维度 ε · L1-K · τ · Top-K
网格组合 1584 (v2)

1实验背景与目标

RAG 检索链由 4 个超参数控制，它们共同决定「问对了能不能查到、问偏了会不会乱答」：

ε
量化步长
向量吸附网格，越大锚点合并越多
L1-K
锚点路由 Top-K
粗筛保留的候选桶数量
τ
相似度阈值
≥τ 判命中，否则建工单
Top-K
精排返回条数
送入 LLM 的上下文条数

核心矛盾：阈值 τ 调低 → 召回高但误命中多（拿不相关知识硬答）；τ 调高 → 误命中少但可能漏答。调优就是找这个平衡点。

2实验设置

超参数候选范围（v2 网格）说明
ANCHOR_QUANT_EPSILON ε0.01 – 0.10（6 档）量化网格步长
RAG_ANCHOR_TOP_K L1-K1, 2, 3, 4, 8, 12, 16, 24锚点粗筛 Top-K
RAG_SCORE_THRESHOLD τ0.35 – 0.85（步长 0.05）命中阈值
RAG_TOP_K Top-K1, 3, 5精排返回条数

评测集构造（v2，共 332 条）：原题 exact 100 + 口语改写 paraphrase 200 + 知识库外 negative 32。v3 进一步扩到 680 条（加 hard_confusion 同类干扰 80 + negative 100），知识库扩至 400 条以测锚点合并。

评测指标

Hit@1_para

主指标——口语改写问题的 Top-1 命中率。衡量「换种问法还能不能查到」。

FPR

误命中率——知识库外问题被误判为命中的比例。越低越好。

L1_recall

锚点召回——正确条目的锚点是否进入 L1 候选桶。验证粗筛不漏。

anchor_compression

锚点压缩比——量化后锚点数 / 条目数，越小索引越轻。

3核心结果

★
一句话结论：最优组合 ε=0.02 · L1-K=8 · τ=0.65 · Top-K=3。其中 τ 从默认 0.4 提到 0.65 是关键——几乎不损召回，却把误命中率从 81% 砍到 3%。

图 1 · 阈值 τ 权衡曲线（ε=0.02, L1-K=8, Top-K=3）

召回 Hit@1_para 与 误命中 FPR 随 τ 的变化
绿线越高越好，红线越低越好。两线的「黄金交叉区」就是最佳阈值。

τ ≤ 0.5 时召回满分但 FPR 高得离谱（0.40 时 81.2%）；τ ≥ 0.75 后召回开始崩塌。τ=0.65 是甜点：Hit_para 仍有 98.5%，FPR 降到 3.1%，满足 ≤5% 约束。

图 2 · 当前默认 vs 推荐配置

把 τ 0.4 → 0.65 的直接效果
召回几乎不变，误命中率断崖式下降。

4结论与生产建议

1. 立即可落地：把 RAG_SCORE_THRESHOLD 从 0.40 调到 0.65，性价比最高。

2. 其余三参数维持默认：ε=0.02、L1-K=8、Top-K=3 已是稳健解，无需改动。

3. 显式写入 .env：当前 RAG_ANCHOR_TOP_K / ANCHOR_QUANT_EPSILON 未写进 .env，靠代码默认兜底，建议显式声明避免歧义。

4. 持续迭代：负样本规模偏小，上线后应收集真实「未命中 / 误命中」日志，定期重跑调优脚本。

推荐 .env

# RAG 最优组合（v3 调优结论）
ANCHOR_QUANT_EPSILON=0.02
RAG_ANCHOR_TOP_K=8
RAG_SCORE_THRESHOLD=0.65   # ← 由 0.40 上调，FPR 81% → 3%
RAG_TOP_K=3
