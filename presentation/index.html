A · RAG 核心 ①

RAG 灵魂：两阶段检索

别人是"一张大表暴力搜"，我们是先粗筛、再精排——又快又准。

L1 · 粗筛路由

kb_anchors 量化锚点

IVFFlat 近似索引快速定位候选桶，取 Top-8 锚点。向量先量化压缩，相似条目共享同一锚点。

→
L2 · 精排

kb_entries 全精度重排

在候选条目上做全精度余弦相似度精排，取 Top-3。score ≥ 0.4 才算命中。

命中 → 交给 DeepSeek 生成答案；未命中 → 进入工单流程。

代码：backend/app/rag/retriever.py · search() 两阶段实现

重点来了，这一页请大家务必看清楚。这是 OpsWarden 的灵魂——两阶段检索。 很多 RAG 系统是怎么做的？一张大向量表，来一个查询就全表暴力搜，数据一多就又慢又不准。我们不一样，我们做了两层。 第一层 L1，叫量化锚点路由。我们用 IVFFlat 近似索引，先快速把范围缩小到 Top-8 个"锚点桶"，相当于先翻到对的章节。 第二层 L2，精排。只在这几个桶里的候选条目上做全精度的余弦相似度计算，取最像的 Top-3，分数要超过 0.4 才算命中。 先粗筛、再精排——这就是又快又准的秘密。命中就交给 DeepSeek 润色生成答案；没命中，就进入工单流程。
A · RAG 核心 ②

把每个细节都打磨到位

BGE-small-zh-v1.5 向量化

中文专用模型，512 维。查询侧加 instruction 前缀（非对称编码），召回更准。

Q + S 联合向量

"问题"和"解决方案"分别编码后取均值，一个向量同时表达两面，检索更全。

量化锚点去重

网格量化 + SHA256 哈希做幂等 upsert，近似重复向量自动合并，索引更轻。

DeepSeek 生成

低温度 0.1，基于检索上下文生成。没 API Key 也能兜底直接返回原文方案。

代码：backend/app/rag/embedder.py · quantizer.py · llm.py

光有架构还不够，魔鬼在细节。这一页是我们 RAG 的四个打磨点。 第一，向量模型选的是 BGE-small-zh，中文专用，512 维。而且我们用了 BGE 的最佳实践——查询侧加 instruction 前缀，做非对称编码，召回率更高。 第二，联合向量。我们把"问题"和"解决方案"分别编码再取均值，一个向量同时表达问题和答案两个侧面，无论用户问法多刁钻，都能检索到。 第三，量化锚点去重。用网格量化加 SHA256 哈希做幂等写入，把近似重复的向量自动合并，索引更轻、检索更快。 第四，生成端用 DeepSeek，温度调到 0.1 保证稳定。而且我们做了兜底——就算没有 API Key，也能直接返回知识库原文，绝不崩。每个细节，我们都抠到位了。

如果说前面是技术，这一页讲的是灵魂里的灵魂——我最骄傲的设计，自学习闭环。 普通的问答机器人，答不上来就是答不上来，永远那么笨。我们的不一样——它会越用越聪明。 看这条闭环：用户提问，RAG 没命中，系统确认后自动建工单，运维同学把问题解决，解决的时候勾一下"写入知识库"，系统就自动把这条新知识向量化、建锚点，存进知识库。 结果是什么？同样的问题，下次再有人问，AI 自己就答出来了，根本不用再开工单。这就叫数字员工——它真的在成长，每解决一个新问题，就永久变聪明一点。这是我们整个系统的闭环，也是它区别于普通问答机器人的关键。
A · 智能体

不止 RAG：LangGraph 智能体

多轮 Agent 编排

  • LangGraph + Postgres 检查点，多轮记忆
  • RAG 只是其中一个工具，共 6 大工具：知识检索 / 工单增改 / 数据统计 / 健康检查
  • 能先查再答，避免重复建单

安全 & 稳健

  • 绝不静默建单：建工单必须用户确认
  • 多层优雅降级：Agent 挂了回退经典管线；模型挂了返回原文
  • 全程审计：agent_runs / tool_calls
我们不止做了 RAG，还往上做了一层 LangGraph 智能体。 左边是能力：基于 LangGraph 加 Postgres 检查点，支持多轮记忆。RAG 在这里只是六大工具之一，还有工单的增改、数据统计、系统健康检查。它能先查再答，比如查一下有没有重复工单，避免乱建单。 右边是我特别想强调的——安全和稳健。第一，绝不静默建单，所有建工单的动作都要用户点确认，不会偷偷给你刷一堆工单。第二，多层优雅降级，智能体挂了就回退到经典 RAG 管线，大模型挂了就直接返回知识库原文，整个系统怎么都不会崩。所有动作还全程审计可追溯。又聪明，又靠谱。
